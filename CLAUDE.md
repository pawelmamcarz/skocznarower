# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Marketing site + bookings app for **skocznarower.pl**, a bicycle service shop in Grodzisk Mazowiecki (PL). Content is Polish. Deployed as a single Cloudflare Worker with the static-assets binding; the Worker handles the apex → www redirect, the booking API, the `/admin` panel, and a daily cron for SMS/email reminders. Static HTML/images fall through to the assets binding.

## Architecture

### Pages (static HTML at the repo root)
- `index.html` — main marketing page. CSS and JS are inlined; do not split into separate files unless asked. Section anchors: `#o-mnie`, `#uslugi`, `#cennik`, `#faq`, `#kontakt`.

The pill stays hidden on error or when nothing is free (a `.next-slot[hidden]{display:none}` rule backs the `hidden` attribute; do not remove it). When the API returns `{date, time}`, the pill links to `/umow?date=YYYY-MM-DD&slot=HH:MM` and `umow.html` reads those query params to preselect the date and slot (format-validated, slot only if still free).

### Design tokens (inline CSS, duplicated across the 6 pages)
There is no shared stylesheet: every page inlines its own `<style>` with a `:root` token block, so tokens are **copied into all 6 HTML files (index, umow, the 3 SEO landings, warsztaty) and must be kept in sync**. The core palette/fonts (`--bg`/`--bg2`/`--bg3`, `--accent` `#9fe22e`, `--accent-dim`, `--white`, `--border`, `--fd` Bebas Neue, `--fb` DM Sans) are mirrored on every page. Three tokens were unified after drifting apart and are now the canonical values everywhere:
- `--radius: 2px` — corner radius for the button family (`.btn`/`.btn-green`/`.nav-cta`/`.btn-submit`/`.reminder-input`/`.map-wrap`). Previously `2/4/6/8px` across files; the homepage's sharp `2px` is the brand canon.
- `--accent-hi: #b5f040` — the brighter green used on `:hover` of accent surfaces. Previously a mix of `#b5f040` (index) and `#b6f23d` (subpages).
- `--muted: #969696` — secondary/label text, chosen to pass WCAG AA (≥4.5:1) on the darkest background it sits on (`--bg3 #1c1c1c`). Previously `#777` (index, failed AA) vs `#888` (subpages).
When changing a button radius, the hover green, or muted text, edit the **token** in all 6 `:root` blocks; do not hardcode a hex or reintroduce raw `#b5f040`/`#b6f23d`/`#777`/`#888`. The token set is deliberately thin: spacing, shadow, and font-size stay as raw magic numbers, not tokens.

Two `index.html`-only mechanics tied to the above: (1) the homepage nav is transparent and `position:fixed` over the hero, offset down by `var(--banner-h, 0)`; nothing sets `--banner-h` since the vacation banner's removal (2026-07-10), so it resolves to 0, but the hook stays for any future top banner (see the vacation banner pattern above); `.nav.stuck` resets it to `top:0` once scrolled. (2) The nav collapses on mobile to a `.nav-toggle` hamburger that toggles `.nav-links.open` (full-width dropdown); without it the section anchors are unreachable on phones. Since 2026-07-10 the hamburger exists on **all 6 pages** (ported from index during the UX-consistency pass), not just the homepage: index collapses at 820px, the 5 subpages at their existing 720px breakpoint, all sharing the same markup (`#nav-toggle` + `#nav-links`), CSS pattern, and Escape/click-to-close JS. The fixed `.mobile-bar` CTA is mobile-only via the `@media(max-width:820px)` rule (`display:none` otherwise), so never give it an inline `display:flex` (it leaks onto desktop and covers the hero pill + footer).

### Worker (`src/index.js`)
- Routed to both `skocznarower.pl/*` and `www.skocznarower.pl/*`. Apex requests get a 301 to `www`. Both routes must stay: drop `skocznarower.pl/*` and apex requests never reach the Worker (the 301 never fires); drop `www.skocznarower.pl/*` and `www` traffic stops being handled.
- `wrangler.jsonc` sets `assets.run_worker_first: true`, so the Worker sees every request first. `/api/*`, `/admin*`, and `/r` are handled in the Worker; everything else falls through to a plain `env.ASSETS.fetch(request)` passthrough. The extensionless resolution (e.g. `/umow` → `umow.html`) is done by the Cloudflare ASSETS binding itself, not by code in `src/index.js`, so do not go hunting for that routing logic in the Worker.
- **Sources of truth that mirror the UI** live as constants at the top of `src/index.js`:
  - `SERVICES` — service list and price labels (14 entries; this is the bookable-form whitelist). Gotcha: `umow.html` carries its **own** copy of `SERVICES` (and `BIKE_TYPES`) as JS arrays near the top of its `<script>`, rendered as a radio-group (`#services`, `renderChoices`), **not** a `<select>`. So editing `SERVICES` in `src/index.js` alone does **not** update the form; ids and prices must match across both, while `umow.html` deliberately uses shortened display labels (e.g. `Odbiór i odwóz roweru` vs `...(adres podaj w notatce)`, `Wymiana części` vs `...(klocki, linki, dętka...)`, `Inne (opisz niżej)` vs `Inne (opisz w notatce)`). Also keep in sync: the visible cennik in `index.html` (`.price-row` sekcja; a finer-grained breakdown of ~26 line items, so the `SERVICES` names appear inside it but the cennik adds sub-items), **i** dodatkowe powierzchnie marketingowe: `serwis-rowerow-pruszkow.html` (Service+OfferCatalog JSON-LD + visible cennik list + FAQPage answer + visible FAQ), `serwis-rowerow-milanowek.html` (te same 4), `bleeding-hamulcow-shimano.html` (Offer + FAQPage), `llms-full.txt` (sekcje `## Pełna lista usług` i `## Cennik`) oraz `llms.txt` (krótkie podsumowanie usług/cen w intro). Notification text in `src/index.js` (SMS reminder/review templates, the seasonal email blast in `sendSeasonalReminders`) currently does **not** quote prices, but if you add a price to any of them, keep it in sync here too.
  - `BIKE_TYPES` — bike-type whitelist for the form (also duplicated as its own array in `umow.html`, alongside that page's copy of `SERVICES`; see the gotcha above).
  - `SCHEDULE` — opening-hours map keyed by day-of-week (0=Sun…6=Sat). Changes here change the available slots returned by `/api/availability`. Synchronizuj też `openingHoursSpecification` w LocalBusiness JSON-LD na `index.html` (Google Knowledge Panel; `closes` = ostatni slot, obecnie 19:00), widoczną sekcję Godziny w `#kontakt` na `index.html` ("Codziennie 16:00 do 19:00"), linię godzin nad polem daty w kroku "03 / Termin" w `umow.html` ("Przyjmuję codziennie... 16:00-19:00"), sekcję godzin w `llms-full.txt` oraz listę slotów w `llms.txt`.
  - `PUBLIC_PHONE_DISPLAY` — the public-facing phone (`600 370 810`, the owner Mateusz's cell, same number as WhatsApp) used in SMS/email copy. Customers call this number directly. History: it was originally the Telnyx virtual number `22 181 15 07` (voice agent never worked in production), then switched to Piotrek the serwisant's cell `501 174 195`, then unified onto `600370810` on 2026-07-07 (decision: one number for both calls and WhatsApp, simpler for customers). It is **not** a separate concept from `OWNER_PHONE` (the owner-alert cell, same default `600370810`) or the WhatsApp `wa.me` number by coincidence of value only, they remain distinct constants that happen to share a default today; changing one does not change the other. When the public number changes, update this constant **and** every page's visible phone, `tel:` links, JSON-LD `telephone`, `llms.txt`/`llms-full.txt`. Note: prior to 2026-07-07, `600370810` was shown on the site **only** as a `wa.me` WhatsApp link, never as visible digits or a `tel:` link; that restriction is now lifted, `600370810` is the single public contact number for calls and WhatsApp alike. In coexistence mode it is also the WhatsApp Cloud API sender, see the WhatsApp channel below.
  Touching any of these means re-checking wszystkie powierzchnie powyżej. Nie dodawaj produktów (np. pakietów) do landingów/llms-full bez wpisu w `SERVICES`, bo klient nie zarezerwuje tego w formularzu.
- New-booking alerts: on each `POST /api/bookings` the owner gets an SMS (to `OWNER_PHONE`, fallback `600370810`) plus the owner email (`NOTIFY_EMAIL`); both fail soft, in isolated try/catch blocks so an owner-side failure never blocks the customer confirmation email. The customer confirmation email (only sent if they gave an email) sets `reply_to` to `REPLY_TO_EMAIL` (falling back to `NOTIFY_EMAIL`) so replies land in a real inbox instead of the `@skocznarower.pl` Cloudflare Email Routing chain. The booking stays `pending` until Mateusz confirms.
- Seasonal-reminder signup (`POST /api/reminders` → `apiSeasonalReminder`): the public end-of-season opt-in. Takes `{email, consent: true}`, validates the email and the consent flag, and inserts the address into `seasonal_reminders` (a duplicate is treated as success, "już jesteś na liście"). The other way rows enter that table is the **`/umow` opt-in**: `createBookingCore` does an `INSERT OR IGNORE` when a booking carries an email and a checked consent checkbox (added 2026-06-30, so paying customers land on the list, not only standalone-form signups). The March 15 cron blast (below) then mails everyone in it. Neither is a cron trigger.
- Voice channel (`/api/voice/*`): server-to-server endpoints an external voice-AI platform (Telnyx) calls as "tools" so an AI agent on a phone number can take bookings. Note: the Telnyx voice agent never worked in production, so the public number was switched to Piotrek's cell (`501 174 195`), then unified onto the owner's cell `600 370 810` on 2026-07-07 (see `PUBLIC_PHONE_DISPLAY` above); these routes remain in the code but are dormant (and gated off by default, see below). The agent's ready-to-paste config lives in `telnyx-voice-agent.md` (dev-only, `.assetsignore`d). The whole namespace is gated by a shared secret `VOICE_API_SECRET` (constant-time compared via `timingSafeEqual`); if the secret is unset every route returns 401, so the channel is off by default and production behavior is unchanged. Routes: `GET /api/voice/availability?date=` (free slots only), `GET /api/voice/next-slot` (reuses `apiNextSlot`), `GET /api/voice/config` (returns `SERVICES`/`BIKE_TYPES`/`SCHEDULE` so the agent prompt stays in sync with the sources of truth), `POST /api/voice/bookings`. Booking creation goes through the shared `createBookingCore` (same validation, conflict checks, `idx_bookings_active_slot` race backstop, and owner SMS+email as the web form), so phone bookings land as `pending` and behave identically; the voice endpoint just adds a `[tel]` notes prefix and a TTS-friendly `confirmation` string. `OWNER_PHONE` (owner alert target) is unrelated to the public/virtual number and stays on the cell.
- WhatsApp channel (`/api/whatsapp/webhook` + `sendWhatsApp`): 600370810 runs the WhatsApp Business **app** and, after coexistence onboarding, also the Cloud API (Meta Graph by default; a BSP like 360dialog is Graph-compatible). `sendWhatsApp(env, phone, message)` sends text/template messages, fail-soft like `sendSms` (without `WHATSAPP_TOKEN`/`WHATSAPP_PHONE_NUMBER_ID` it logs `[WA dry-run]` and production is unchanged). Outbound is wired into `sendNotifications` (customer booking confirmation, template `potwierdzenie_rezerwacji`, alongside the email) and `sendDailyReminders` (24h reminder, template `przypomnienie_wizyty`, with SMS as fallback); the review request in `sendFollowUps` stays on SMS. Inbound: `GET /api/whatsapp/webhook` is the Meta verify handshake (returns `hub.challenge`, token compared constant-time to `WHATSAPP_VERIFY_TOKEN`, fails closed with 403 if unset); `POST` verifies `X-Hub-Signature-256` (HMAC-SHA256 over `WHATSAPP_APP_SECRET` via `hmacHex`) and likewise fails closed with 403 if `WHATSAPP_APP_SECRET` is unset (same posture as `VOICE_API_SECRET`, no silent skip), then logs/stores messages in `whatsapp_messages`, always returns 200, and can auto-reply if `WHATSAPP_AUTO_ACK=1` (off by default, since in coexistence the owner replies in the app). This route is **outside** the `VOICE_API_SECRET` gate (Meta calls it publicly). Template names/language are overridable via `WHATSAPP_TPL_CONFIRM`/`WHATSAPP_TPL_REMINDER`/`WHATSAPP_LANG`. Owner-side onboarding (Meta verification, coexistence QR, template approval, secrets) is documented in `whatsapp-setup.md` (dev-only, `.assetsignore`d).
- Quick action from the SMS: the SMS carries a link `/r?id=<id>&t=<token>` where the token is a truncated (24-char) HMAC of the string `r:<id>` over the session secret (`bookingToken`). `GET /r` renders a small page (booking details + Potwierdź/Odrzuć buttons); the action runs only via `POST /r`, so SMS link scanners can't trigger it. The token never expires and isn't one-time-use, so both GET and POST re-check the booking's current status first (shared `alreadyDecided` check) and show an "already confirmed/cancelled/done" message instead of acting, in case the status changed since the link was loaded (e.g. a stale, unrefreshed tab). Confirm/cancel reuse the shared `confirmBooking`/`cancelBooking` helpers (same path as `/admin`), so confirming via the link also creates the calendar event.
- Google Calendar: confirming a booking (via `/admin` `action=confirm` **or** the SMS quick-action `POST /r`, both call the shared `confirmBooking`) creates a Calendar event in the "pyszczka" calendar and stores its id in `bookings.gcal_event_id`; cancelling/deleting removes the event. Auth is a Google service account (JWT bearer, RS256 via Web Crypto) needing `GOOGLE_SA_EMAIL`, `GOOGLE_SA_PRIVATE_KEY`, `GOOGLE_CALENDAR_ID` (calendar must be shared with the service-account address). Without these it logs a dry-run.
- Admin auth: login authenticates against `ADMIN_PASSWORD` only (single shared password, no per-user accounts; Piotrek the serwisant and Mateusz use the same one); the session cookie is HMAC-signed with `SESSION_SECRET` (falling back to `ADMIN_PASSWORD` if unset). `ADMIN_PASSWORD` is therefore required to log in, with only `SESSION_SECRET` set you cannot authenticate. The `/admin` page lists bookings, lets you set final price, mark done/cancel, and block slots/days in `blocked_slots`.
- Booking lifecycle in `/admin` (`POST /admin/booking`, action-based, all via `adminUpdateBooking`): `confirm` (→ confirmed + calendar; also serves as "Przywróć" for cancelled rows), `start` (→ `in_progress`, "przyjęto do serwisu", sends the customer an SMS via `repairAcceptedSms`), `done` (→ done, sends an SMS summary via `repairDoneSms`, and stores the optional `repair_summary` free-text typed in the row), `cancel`, `delete`, `price`. The two lifecycle SMS only fire on a real status transition (guarded by reading the row's prior status), so re-clicking does not re-send; they go out inline (awaited, not via cron). `repairDoneSms` quotes `repair_summary` if set else the service name, plus `final_price` if set; the dashboard "Zrobione" form also submits the current value of the row's price input (JS copies it to a hidden `final_price` before submit), so a price typed but not yet saved is not lost. `start`/`done` are guarded against resurrecting cancelled rows (use Przywróć first) and map `idx_bookings_active_slot` UNIQUE collisions to a toast error. Post-action UX: every action redirects with `?msg=`/`?err=` codes mapped through the `ADMIN_MSGS`/`ADMIN_ERRS` whitelists in `src/index.js` (raw query values never reach HTML); destructive and SMS-sending actions have `confirm()` dialogs; each action is its own `<form>` so Enter inside the "co zrobiono" input never submits anything. The dashboard opens with a "Dziś" section (all of today's bookings regardless of status) and the pending counter on the Nadchodzące tab is a global COUNT, not filter-scoped. Przeszłe/Wszystkie sort newest-first. Below 720px the admin tables render as cards (`table.cards` + `td[data-label]`) with 44px touch targets. Status `in_progress` ("w naprawie") counts as active for `idx_bookings_active_slot` and shows under the **Nadchodzące** filter. Blokada terminów przyjmuje zakres dat (Od / Do opcjonalnie, `INSERT OR IGNORE` po dniach, limit 60 dni, `time_slot='all'`), a ciągłe zablokowane dni sklejają się w tabeli w jeden wiersz z jednym Odblokuj; pola godzin w panelu to `<input type="time" step="3600">` z serwerową walidacją HH:MM.
- Manual booking entry (`POST /admin/booking-new` → `adminCreateBooking`): the "Dodaj rezerwację ręcznie" card lets Piotrek/Mateusz log phone or Google-Places customers (incl. an optional `bike_model`). Deliberately looser than the public form: it allows past dates and off-`SCHEDULE` times, sends no notification on create (the lifecycle SMS above cover that), inserts directly as `confirmed`, prefixes notes with a source tag (`[tel]`/`[google]`/`[ręczna]`), and adds a calendar event only for today-or-future dates. Duplicate active slot → 409, bad whitelist/field → 400.
- Per-order detail page "zlecenie" (the row "Klient" name and an "Otwórz" button both link to `GET /admin/zlecenie?id=`, rendered by `adminBookingDetail`). It is the full repair-order view: sekcje **Terminy** (`accepted_at` termin przyjęcia — auto-stamped on the `start` action, manually overridable via the `accepted_date` date field; `expected_ready_date` oczekiwany odbiór), **Naprawa i notatki** (the client's `notes` shown read-only, an editable warsztat field `repair_info`, the short `repair_summary` that feeds the done-SMS, `bike_model`), **Ceny i płatność** (the settlement fields below + `final_price_override`), a collapsed `<details>` **Podział Mateusz/Piotr**, and a **Komunikacja z klientem** section. `POST /admin/zlecenie` is dispatched by `adminZleceniePost`: `action=finance` → `adminSaveFinance(env, form)` (one form saves all of the above; a save writes every field so the form always submits all of them), `action=send_sms` → `adminSendSms` (sends a free-text SMS via `sendSms` and logs it to the `messages` table). Accepted-date preservation: if the submitted date equals the stored `accepted_at`'s Warsaw date the precise ms is kept, else `dateToMs` stores noon-UTC ms of the new date.
- Communication timeline (`buildTimeline`): reconstructs **outbound lifecycle SMS from timestamps** (`accepted_at`, `done_at`, `reminder_sent_at`, `feedback_sent_at` — these are "wysłany", not delivery-confirmed, since `sendSms` is fire-and-forget and unlogged), plus manual panel SMS from `messages`, plus inbound/outbound WhatsApp from `whatsapp_messages` matched by `waPhoneKey(customer_phone)` (labelled "po numerze", may catch the same customer's other bookings). The `messages` table (migration 0012) only logs panel-sent SMS for now; a full provider message log + panel WhatsApp is future work.
- Mateusz/Piotr settlement math (`computeSettlement`): the effective client price `total = final_price_override ?? (parts_charged + labor_charge)`, and **zysk = total − parts_cost** (so a flat override / discount also splits), divided **75% Mateusz / 25% Piotr** (`SPLIT_MATEUSZ`, `Math.round` on Mateusz, Piotr = remainder); if `service_by='mateusz'` it is a solo job → **100% Mateusz, 0 Piotr**. Without an override this equals the original `(parts_charged − parts_cost) + labor_charge`. The "zapłacono ≠ wycena" warn and the split box compare against this effective `total`. `parts_cost` is a pass-through reimbursement to the `parts_by` buyer. The cash holder (`paymentHolder`) is `paid_to` if set else cash→Piotr, blik/transfer→Mateusz; net per person = collected − owed, positive = holds surplus and pays the other. `final_price` (used by the done-SMS "Koszt" and dashboard revenue) = `final_price_override` if set, else `parts_charged + labor_charge` when either is provided. The **Rozliczenie** page (`GET /admin/rozliczenie` → `adminSettlement`) lists `done` jobs with per-order split and an aggregate "kto komu ile przekazać" (transfer bounded to `min(surplus, deficit)` so customer over/underpayment doesn't skew it); `POST /admin/rozliczenie` action=settle/unsettle toggles `settled_at` (settled jobs hide from the default view, visible under `?show=all`). **The Rozliczenie nav link is currently hidden** (commented out in both top bars per owner request); the route and the detail-page split box still work. Amounts are whole-złoty INTEGERs.
- A second tab **Współpraca** (anchor `#outreach`) is an outreach tracker over `outreach_contacts`: add/send/respond/close/reopen, with two independent filters, status (planned/sent/responded/closed) and channel (A = brand/dealer, B = dystrybutor/program serwisowy, C = sklep bez warsztatu). Routes: `POST /admin/outreach` (action-based, redirects to `/admin#outreach`).
- `scheduled()` runs on the `0 8 * * *` cron (08:00 UTC ≈ 09:00–10:00 Warsaw depending on DST):
  - SMS reminder 24h before each `confirmed` booking (sets `reminder_sent_at`).
  - SMS asking for a Google review 3 to 30 days after a `done` booking (selection is `date <= -3d AND date >= -30d AND feedback_sent_at IS NULL`; sets `feedback_sent_at`). The link comes from the `REVIEW_LINK` var (a `wrangler.jsonc` var, not a secret); the SMS falls back to the homepage if it is unset, while `/api/reviews` exposes `REVIEW_LINK` only when it is not the `CHANGE_TO_` placeholder.
  - **Win-back SMS** (`sendWinBack`, migration 0013): to customers whose most recent visit was 6 to 18 months ago, who did not return and have a `done` job. Gated off by default, runs only when `WINBACK_ENABLED === '1'` (production unchanged until you flip it). Eligibility is evaluated against each phone's single most-recent active booking (picked via `ROW_NUMBER()`, ties broken by `id`, so a same-day double booking never double-sends), `ORDER BY date ASC` + `LIMIT 25` per run (trickles, never a blast, deterministic day-by-day), excludes anyone with a future booking (their latest `date` exceeds the cutoff), and stamps `winback_sent_at` only on that one latest-booking row, not the whole phone's history, so a customer becomes re-eligible once they visit again and lapse a second time.
  - Seasonal email blast on March 15 to everyone in `seasonal_reminders`.
  - Google reviews refresh: pulls up to 5 reviews from Google Places API (New) into `google_reviews` + `google_profile`. `/api/reviews` returns the cache as JSON (up to 6 rows, newest first), `index.html` loads it client-side and shows the `#opinie` section only if reviews exist. Manual refresh button on `/admin`.
  Sending providers: SMS via SMSAPI (`api.smsapi.pl/sms.do`, OAuth bearer token), email via Resend. Reviews via `places.googleapis.com/v1/places/{id}` with `X-Goog-Api-Key`. All three fail soft — without their secrets the Worker logs a dry-run and continues.

### `.assetsignore` (what does NOT get uploaded as a static asset)
The Worker bundles the whole repo root as assets, so this file is what keeps junk out: large source media (`uploads/*.mp4`, `uploads/*.pdf`, `uploads/FastDL.to_*`), the Worker source (`src/`), migrations, dotfiles (`.wrangler/`, `.git/`, `.claude/`, `.playwright-mcp/`, `.DS_Store`), `node_modules/`, the PDF sources (`broszura-warsztaty.source.html`, `voucher-warsztaty.source.html`), `partners/` (offline business docs, e.g. a partner survey .doc; never meant to be served), and the dev-only files (`CLAUDE.md`, `OUTREACH_PLAN.md`, `WARSZTATY_PLAN.md`, `telnyx-voice-agent.md`, `whatsapp-setup.md`, `.assetsignore`, `.gitignore`, `.dev.vars*`). When adding new top-level files or directories, decide whether they belong in `.assetsignore` — `wrangler deploy` uploads from disk, so even git-untracked directories get published unless listed here.

### `_headers` (cache-control for static assets)
`_headers` at the repo root is a Cloudflare assets-config file (consumed by the ASSETS binding, **not** served as a static asset, so it stays **out** of `.assetsignore`). It sets `/uploads/*` to `Cache-Control: public, max-age=31536000, immutable` so images/favicons/the hero poster cache for a year in the browser instead of revalidating every visit (without it the binding defaults to `max-age=0, must-revalidate`, i.e. a conditional request per asset on every repeat visit). HTML is deliberately left on the default so content edits go live immediately. It applies through the `env.ASSETS.fetch` passthrough even with `run_worker_first: true`. Filenames in `uploads/` are stable, so `immutable` is safe; if you ever overwrite an image in place rather than adding a new filename, bump the filename (or the cache will serve the old bytes for up to a year).

## Content conventions (load-bearing)

These come from explicit user preference, not house style — apply them to every text change:

1. **No em-dashes (—).** Use a comma, semicolon, colon, or rewrite. This includes JSON-LD answer strings, meta descriptions, visible copy, and any SMS/email text in `src/index.js`.
2. **Avoid "I-slope" narration.** No Polish sentences starting with "Ja" or leaning on first-person interjections as a stylistic device. Keep copy direct and second-person where possible.
3. **No suspension service (decision 2026-07-02).** The shop does NOT service suspension: never list "zawieszenie", "amortyzatory", damper/fork service, sag/kompresja/odbicie tuning, or suspension brands (RockShox, Fox, Manitou, DVO, X-Fusion) as an offered service anywhere (visible copy, JSON-LD, meta descriptions, llms files, SMS). "Full suspension" / "MTB z zawieszeniem" may appear only as a *bike type* the shop accepts (e.g. "Hardtail i full suspension. Hamulce, koła, napęd."), never as a serviced component.

Before committing any copy change (HTML or Worker strings), re-scan the diff for `—` characters.
